1 Project Setup & API Enablement ~3 min

1.1 Create config.env

cd deploy/gcp
cp config.env.template config.env
# Edit config.env with your GCP_PROJECT_ID

1.2 Create GCP Project

gcloud projects create agentshield-poc --name="AgentShield POC"

1.3 Link Billing

# Find your billing account ID
gcloud billing accounts list

# Link it
gcloud billing projects link agentshield-poc \
  --billing-account=YOUR_BILLING_ACCOUNT_ID

1.4 Set Active Project & Region

gcloud config set project agentshield-poc
gcloud config set run/region us-central1
gcloud config set compute/region us-central1
⚠️ Bottleneck: Compute API Prompt

Setting compute/region triggers a prompt: "API [compute.googleapis.com] not enabled. Enable? (y/N)". Type y and wait ~1 minute. This is expected on a fresh project.

1.5 Enable All Required APIs

gcloud services enable \
  run.googleapis.com \
  sqladmin.googleapis.com \
  redis.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com \
  cloudbuild.googleapis.com \
  vpcaccess.googleapis.com \
  firebasehosting.googleapis.com \
  firebase.googleapis.com \
  --project=agentshield-poc
ℹ️ Tip

Enable firebasehosting.googleapis.com and firebase.googleapis.com here too — saves a round-trip later in Step 5.

2 Infrastructure (Cloud SQL + Redis + VPC) ~10 min

2.1 Generate DB Password

DB_PASSWORD=$(openssl rand -base64 24 | tr -d '/+=' | head -c 24)
echo "Password: $DB_PASSWORD"
# Save this in config.env → DB_PASSWORD=...

2.2 Create Cloud SQL Instance

🚨 Critical: Correct Tier Name

Do NOT use db-f1-micro — this fails with "Invalid Tier for ENTERPRISE_PLUS Edition". Use db-custom-1-3840 with --edition=enterprise instead.

gcloud sql instances create agentshield-db \
  --database-version=POSTGRES_16 \
  --tier=db-custom-1-3840 \
  --edition=enterprise \
  --region=us-central1 \
  --storage-size=10GB \
  --storage-type=SSD \
  --availability-type=zonal \
  --assign-ip \
  --authorized-networks=0.0.0.0/0 \
  --project=agentshield-poc \
  --quiet

# ⏱️ Takes 3-5 minutes

2.3 Create VPC Connector (run in parallel with SQL)

gcloud compute networks vpc-access connectors create agentshield-vpc \
  --region=us-central1 \
  --range=10.8.0.0/28 \
  --project=agentshield-poc \
  --quiet

# ⏱️ Takes 2-3 minutes

2.4 Create Memorystore Redis

gcloud redis instances create agentshield-redis \
  --size=1 \
  --region=us-central1 \
  --redis-version=redis_7_0 \
  --project=agentshield-poc \
  --quiet

# ⏱️ Takes 5-7 minutes
ℹ️ Parallelization Tip

Cloud SQL, VPC connector, and Redis have no dependencies. Run all three in separate terminal tabs to save ~10 minutes.

2.5 Create Database & User (after Cloud SQL is ready)

# Create database
gcloud sql databases create agentshield \
  --instance=agentshield-db \
  --project=agentshield-poc

# Create user
gcloud sql users create agentshield \
  --instance=agentshield-db \
  --password="$DB_PASSWORD" \
  --project=agentshield-poc

2.6 Note Redis IP (after Redis is ready)

REDIS_HOST=$(gcloud redis instances describe agentshield-redis \
  --region=us-central1 --project=agentshield-poc \
  --format='value(host)')
echo "Redis IP: $REDIS_HOST"
# Our deployment: 10.20.57.35

3 Secret Manager ~1 min

3.1 Create Secrets

# Generate secrets
JWT_SECRET=$(openssl rand -hex 32)
COMPLIANCE_KEY=$(openssl rand -base64 32 | head -c 32)

# Store in Secret Manager
echo -n "$JWT_SECRET" | gcloud secrets create agentshield-jwt-secret \
  --data-file=- --project=agentshield-poc

echo -n "$DB_PASSWORD" | gcloud secrets create agentshield-db-password \
  --data-file=- --project=agentshield-poc

echo -n "$COMPLIANCE_KEY" | gcloud secrets create agentshield-compliance-key \
  --data-file=- --project=agentshield-poc

3.2 Grant Cloud Run Access

PROJECT_NUMBER=$(gcloud projects describe agentshield-poc \
  --format='value(projectNumber)')
SA="${PROJECT_NUMBER}-compute@developer.gserviceaccount.com"

for secret in agentshield-jwt-secret agentshield-db-password agentshield-compliance-key; do
  gcloud secrets add-iam-policy-binding "$secret" \
    --member="serviceAccount:$SA" \
    --role="roles/secretmanager.secretAccessor" \
    --project=agentshield-poc \
    --quiet
done

DB_NAME=agentshield,\

DB_SSL=false,\

REDIS_PORT=6379,\

ADMIN_PASSWORD=admin123" \

# Subsequent: 1-2 min (cached layers)

4.2 Verify Health

BACKEND_URL=$(gcloud run services describe agentshield-api \
  --region=us-central1 --project=agentshield-poc \
  --format='value(status.url)')

curl -s "$BACKEND_URL/health" | python3 -m json.tool
# Should show: {"status":"healthy","database":{"status":"healthy"}}

4.3 Run Migrations

# Get the deployed image
IMAGE=$(gcloud run services describe agentshield-api \
  --region=us-central1 --project=agentshield-poc \
  --format='value(spec.template.spec.containers[0].image)')

# Create migration job
gcloud run jobs create agentshield-migrate \
  --image="$IMAGE" \
  --region=us-central1 \
  --project=agentshield-poc \
  --set-cloudsql-instances=agentshield-poc:us-central1:agentshield-db \
  --vpc-connector=agentshield-vpc \
  --set-env-vars="DB_HOST=/cloudsql/agentshield-poc:us-central1:agentshield-db,\
DB_NAME=agentshield,DB_USER=agentshield,DB_SSL=false" \
  --set-secrets="DB_PASSWORD=agentshield-db-password:latest" \
  --command="npm" --args="run,migrate" \
  --max-retries=0 --quiet

# Execute
gcloud run jobs execute agentshield-migrate \
  --region=us-central1 --project=agentshield-poc --wait
⚠️ Bottleneck: Cloud Run Jobs Flag Difference

Cloud Run services use --add-cloudsql-instances. Cloud Run jobs use --set-cloudsql-instances. Using the wrong flag throws an unrecognized argument error.

4.4 Seed Admin User

🚨 Critical Fix: Invalid Bcrypt Hash in Migration

The migration file 001_initial_schema.sql seeds an admin user with an invalid bcrypt hash that doesn't correspond to any password. Login will fail with "Invalid email or password" even though the user exists in the database.

Solution: The src/db/seed.js script was created to force-update the password hash. It must run after migrations.

# Create seed job (uses same image as migration)
gcloud run jobs create agentshield-seed \
  --image="$IMAGE" \
  --region=us-central1 \
  --project=agentshield-poc \
  --set-cloudsql-instances=agentshield-poc:us-central1:agentshield-db \
  --vpc-connector=agentshield-vpc \
  --set-env-vars="DB_HOST=/cloudsql/agentshield-poc:us-central1:agentshield-db,\
DB_NAME=agentshield,DB_USER=agentshield,DB_SSL=false,\
ADMIN_EMAIL=admin@agentshield.local,ADMIN_PASSWORD=admin123" \
  --set-secrets="DB_PASSWORD=agentshield-db-password:latest,\
JWT_SECRET=agentshield-jwt-secret:latest" \
  --command="npm" --args="run,seed" \
  --max-retries=0 --quiet

gcloud run jobs execute agentshield-seed \
  --region=us-central1 --project=agentshield-poc --wait

5 Deploy Dashboard to Firebase Hosting ~3 min

5.1 Code Change Required

The dashboard hardcodes API_BASE to localhost:3000. One line change makes it configurable:

// File: agentshield-dashboard/src/api.js — Line 1
- const API_BASE = 'http://localhost:3000/api/v1';
+ const API_BASE = import.meta.env.VITE_API_BASE || 'http://localhost:3000/api/v1';
✅ Backwards Compatible

This change falls back to localhost:3000 if VITE_API_BASE is not set, so local development is unaffected.

5.2 Build Dashboard with Production API URL

cd agentshield-dashboard

VITE_API_BASE="$BACKEND_URL/api/v1" npm run build

5.3 Initialize Firebase & Create Hosting Site

🚨 Critical: Firebase Must Be Initialized First

You cannot deploy to Firebase Hosting without first adding Firebase to the GCP project and creating a hosting site. Attempting to deploy directly will fail with: "Unable to deploy — no Hosting site" or "Assertion failed: resolving hosting target".

# Step 1: Add Firebase to the GCP project
npx -y firebase-tools projects:addfirebase agentshield-poc

# Step 2: Create a hosting site
npx -y firebase-tools hosting:sites:create agentshield-dashboard \
  --project agentshield-poc

5.4 Configure & Deploy

# Create firebase.json
cat > firebase.json << 'EOF'
{
  "hosting": {
    "site": "agentshield-dashboard",
    "public": "dist",
    "ignore": ["firebase.json", "**/.*", "**/node_modules/**"],
    "rewrites": [{ "source": "**", "destination": "/index.html" }]
  }
}
EOF

# Create .firebaserc
cat > .firebaserc << 'EOF'
{ "projects": { "default": "agentshield-poc" } }
EOF

# Deploy
npx -y firebase-tools deploy --only hosting:agentshield-dashboard \
  --project agentshield-poc
ℹ️ Site URL

Dashboard will be available at: https://agentshield-dashboard.web.app

🚨 Critical Fix: Dockerfile Must Use npm install

The agent Dockerfiles originally used npm ci, which requires a package-lock.json. Since these are lightweight single-file agents without lockfiles, the build fails silently with "Build failed; check build logs for details".

Solution: Change Dockerfiles to use npm install --production:

# Correct Dockerfile for all test agents:
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install --production
COPY . .
EXPOSE 8080
CMD ["node", "server.js"]

7 Register Agents in AgentShield ~1 min

Login & Register

API="$BACKEND_URL/api/v1"

# Login
TOKEN=$(curl -s -X POST "${API}/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@agentshield.local","password":"admin123"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['data']['token'])")

# Register each agent via /api/v1/agents (NOT /admin/agents)
curl -s -X POST "${API}/agents" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Echo Agent",
    "slug": "echo-agent",
    "type": "internal",
    "protocol": "rest",
    "endpointUrl": "https://echo-agent-PROJECT_NUMBER.us-central1.run.app",
    "description": "Baseline echo agent"
  }'

# Repeat for finance-agent, support-agent, weather-mcp
# Note: weather-mcp endpoint is /mcp/sse (the SSE transport endpoint)
⚠️ Bottleneck: Correct API Route

Agent registration uses POST /api/v1/agents (the registry routes), NOT /api/v1/admin/agents. The admin route returns 404.

⚠️ Bottleneck: JWT Token Expiry

JWT tokens expire in 15 minutes. If registration takes longer (e.g., agents have cold starts), re-login before registering. The first invocations hit cold-start delays of 10-30 seconds.

🚨 Critical Fix: Pre-seeded Budgets Block Invocations

The migration seeds 6 demo budget records with pre-filled usage that already exceeds limits (e.g., "Q3 Demo Project" at 110% of budget). This causes all agent invocations to return HTTP 402 "Budget exceeded".

Solution: Delete all demo budgets before testing:

# List budgets
curl -s "${API}/budgets" -H "Authorization: Bearer $TOKEN" | python3 -m json.tool

# Delete each demo budget by ID
curl -s -X DELETE "${API}/budgets/BUDGET_ID" \
  -H "Authorization: Bearer $TOKEN"

🔧 Bottlenecks & Fixes (Critical Reference)

These are the 6 issues encountered during the first deployment, listed in order of severity:

1. 🚨 Invalid Bcrypt Hash in Migration (Login Fails)

SymptomLogin returns "Invalid email or password" despite admin user existing in DB
Cause001_initial_schema.sql line 230 contains a fabricated bcrypt hash that doesn't match any password
FixCreated src/db/seed.js that force-updates the password hash after checking if the user exists
PreventionGenerate valid bcrypt hash: node -e "require('bcryptjs').hash('admin123',10).then(h=>console.log(h))" and replace in migration

2. 🚨 Agent Dockerfiles Use npm ci Without Lockfile

SymptomAll 4 agent builds fail silently: "Build failed; check build logs for details"
CauseDockerfiles had RUN npm ci --production but agents have no package-lock.json
FixChanged to RUN npm install --production in all agent Dockerfiles
PreventionEither generate lockfiles (npm install locally first) or use npm install in Dockerfile

3. 🚨 Pre-seeded Budgets Block All Invocations (HTTP 402)

SymptomAll POST /gateway/agents/*/invoke return 402 "Budget exceeded"
CauseMigration seeds 6 demo budgets with current_cost_cents already exceeding limits
FixDelete all demo budgets via DELETE /api/v1/budgets/:id
PreventionDon't seed usage data in demo budgets, or seed with hard_limit=false

4. ⚠️ Cloud Run Jobs vs Services — Different Flag Names

Symptomgcloud run jobs create fails with "unrecognized arguments: --add-cloudsql-instances"
CauseServices use --add-cloudsql-instances, Jobs use --set-cloudsql-instances
FixUse --set-cloudsql-instances for all gcloud run jobs commands

5. ⚠️ Cloud SQL Tier Name Changed

Symptomgcloud sql instances create fails: "Invalid Tier (db-f1-micro) for ENTERPRISE_PLUS"
CauseGCP deprecated db-f1-micro. Must specify --edition=enterprise with new tier names
FixUse --tier=db-custom-1-3840 --edition=enterprise

6. ⚠️ Firebase Hosting Requires Two-Step Init

Symptomfirebase deploy fails: "no Hosting site" or "Assertion failed: resolving hosting target"
CauseA fresh GCP project has no Firebase project or hosting site
FixRun projects:addfirebase then hosting:sites:create before deploying

📝 Lessons Learned & Best Practices

For Future Deployments

  1. Fix the migration seed data first. Before deploying, update 001_initial_schema.sql with a valid bcrypt hash, or remove the INSERT entirely and rely solely on seed.js.
  2. Always generate lockfiles for Docker builds. Run npm install locally in each agent directory to generate package-lock.json before deploying.
  3. Seed demo data carefully. Don't seed budget records with usage that already exceeds limits. Use current_tokens=0 and current_cost_cents=0.
  4. Enable ALL APIs upfront. Include Firebase APIs in the initial batch to avoid extra round-trips.
  5. Parallelize infrastructure creation. Cloud SQL, Redis, and VPC connector have zero dependencies — run them simultaneously to save 10+ minutes.
  6. Use environment variables, not code changes. The VITE_API_BASE pattern keeps the codebase deployment-agnostic.
  7. Set min-instances=1 for the backend. Cold starts add 10-30 seconds of latency. The backend should always be warm.
  8. Set min-instances=0 for test agents. They only run during testing, so scale-to-zero saves money.
  9. Redis is optional but recommended. The backend works without Redis (skips caching). Connect it later to improve gateway latency.
  10. JWT tokens expire in 15 minutes. If a step takes longer (cold starts, build retries), re-login before the next API call.

# Step 1: Project + APIs (3 min)
gcloud projects create agentshield-poc --name="AgentShield POC"
gcloud billing projects link agentshield-poc --billing-account=BILLING_ID
gcloud config set project agentshield-poc
gcloud services enable run.googleapis.com sqladmin.googleapis.com redis.googleapis.com \
  artifactregistry.googleapis.com secretmanager.googleapis.com cloudbuild.googleapis.com \
  vpcaccess.googleapis.com firebasehosting.googleapis.com firebase.googleapis.com

# Step 2: Infra — run all 3 in parallel (10 min)
gcloud sql instances create agentshield-db --database-version=POSTGRES_16 \
  --tier=db-custom-1-3840 --edition=enterprise --region=us-central1 ... &
gcloud compute networks vpc-access connectors create agentshield-vpc ... &
gcloud redis instances create agentshield-redis --size=1 --region=us-central1 ... &
wait
