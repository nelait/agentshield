00 Prerequisites & Setup

🔧 Environment Requirements
  • AgentShield backend running (local npm start or deployed at GCP)
  • PostgreSQL with migration 009_guardrails.sql applied
  • At least one agent registered (e.g., GPT-4 Analyst via seed data)
  • Logged in as admin or editor role
  • Dashboard accessible at https://agentshield-dashboard.web.app or http://localhost:5173
  • Guardrails module enabled in Settings → Module Toggles
⚠️ Before You Begin Ensure the Guardrails module is enabled in Settings → Module Toggles. If disabled, the gateway middleware will skip guardrail enforcement entirely. Run npm run migrate if the guardrail_profiles table does not exist.

API Base URL

# Local development
BASE_URL=http://localhost:3000/api/v1/admin

# Production (GCP)
BASE_URL=https://<your-cloud-run-url>/api/v1/admin

Authentication Header

# Get token via login
curl -X POST $BASE_URL/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@agentshield.local","password":"admin123"}'

# Use token in subsequent requests
AUTH="Authorization: Bearer <your-jwt-token>"

01 Profile Management (CRUD)

A Guardrail Profile is a named collection of rules that can be assigned to one or more agents. Each profile has a mode that determines enforcement behavior.

Profile Modes
ModeBehavior
blockViolations with critical or high severity will block the request (HTTP 403)
log_onlyViolations are logged and returned but the request is allowed to proceed

04 Test Runner — Guardrail Tests

The Test Runner lets you validate guardrail rules by running test cases with known inputs and expected verdicts (✓ pass or ✗ block).

Test Case ──► Input Text ──► All Profile Rules ──► Actual Verdict vs. Expected Verdict ↓ PASS (match) or FAIL (mismatch)
ℹ️ Test Case Structure Each test case must include:
  • input — The text to evaluate (string)
  • expectedVerdict — "pass" (should be allowed) or "block" (should be blocked)
  • description — A label for the test case
  • direction — "input" or "output" (default: "input")

TC-T04 Run Tests — Topic Boundary Medium ▼

Objective: Verify topic enforcement — allowed and blocked topics.

🔴 Critical: Severity vs. Blocking Behavior The topic_boundary rule below is created with severity: "medium". In block mode, only critical and high severity violations actually block requests. Therefore, even though the topic boundary rule triggers (violations are detected and logged), the actual verdict remains "pass" because the severity is too low to block.

To make topic violations actually block requests, change the rule severity to "high" or "critical". 
curl -X POST $BASE_URL/guardrails/profiles/<PROFILE_ID>/test \
  -H "Content-Type: application/json" \
  -H "$AUTH" \
  -d '{
    "testCases": [

        "input": "What was the Q3 revenue and how does the budget look for next year?",
        "expectedVerdict": "pass",
        "description": "Contains allowed topics: revenue, budget"
      },
      {
        "input": "What are your thoughts on politics and the upcoming election?",
        "expectedVerdict": "pass",
        "description": "Contains blocked topic — but severity=medium so NOT blocked"
      },
      {
        "input": "Tell me about the best gambling strategies for poker.",
        "expectedVerdict": "pass",
        "description": "Contains blocked topic — but severity=medium so NOT blocked"
      },
      {
        "input": "How is the weather today in San Francisco?",
        "expectedVerdict": "pass",

A Rule Type Quick Reference

Rule Type Scope Config Schema Trigger Condition Example Trigger
content_filter input / output / both { keywords: string[], caseSensitive: bool } Text contains any listed keyword "How do I hack this?"
pii_shield input / output / both { patterns: string[], customPatterns: [{pattern, flags, label}] } Text matches SSN, CC, email, phone, DOB, MRN patterns "SSN: 123-45-6789"
prompt_injection input { extraPatterns: string[] } Text matches any of 15+ jailbreak/injection regex patterns "Ignore all previous instructions"
topic_boundary input / output / both { allowedTopics: string[], blockedTopics: string[] } Text mentions blocked topic OR misses all allowed topics "Tell me about politics"
token_limit input / output / both { maxTokens: number } Estimated token count exceeds limit (1 token ≈ 4 chars) 8,200+ character input (≈ 2,050 tokens > 2,048 limit)
custom_regex input / output / both { patterns: [{pattern, flags, label}] } Text matches any custom regex pattern "See PROJECT-ABC-1234"
output_format output { requireJson: bool, maxLength: number } Output is not valid JSON or exceeds max length "Hello world" (not JSON)
