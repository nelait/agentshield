01 Integration Architecture

AgentShield operates as an AI Agent Governance Firewall that sits between consumers (applications, users, other agents) and AI agents. It supports two integration modes depending on your architecture.

MODE 1: Full Gateway (AgentShield proxies everything) Client ──▶ AgentShield Gateway ──▶ Agent │ Auth → Policy → Guardrails → Proxy → Cost → Audit (AgentShield handles the ENTIRE lifecycle) MODE 2: Sidecar / Policy Decision Point (PDP) Client ──▶ Your Gateway (Kong / Apigee / Envoy / AWS API GW) ──▶ Agent │ └──▶ AgentShield /policy/check (returns allow/deny in ~100ms) Your gateway routes traffic only if allowed
Recommended for POC

Mode 1: Full Gateway

AgentShield acts as the single entry point. All traffic flows through it, enabling every governance feature: auth, policies, guardrails, cost tracking, and audit logging.

Your existing API gateway routes traffic. It calls AgentShield only for policy decisions. Best when you already have Kong, Apigee, Envoy, or AWS API Gateway.

02 Mode 1 — Full Gateway

In Full Gateway mode, clients call AgentShield instead of the agent directly. AgentShield authenticates, enforces policies, scans inputs/outputs, proxies the request, tracks costs, and logs everything.

URL Pattern

POST https://agentshield-api-622662891364.us-central1.run.app/api/v1/gateway/agents/{agent-slug}/invoke

Authentication

Every gateway call requires authentication via one of:

MethodHeaderExample
JWTAuthorization: Bearer {token}Obtain via POST /auth/login
API KeyX-API-Key: {key}Create via Settings → API Keys

03 Mode 2 — Sidecar / Policy Decision Point (PDP)

In sidecar mode, your existing API gateway (Kong, Apigee, Envoy, AWS API Gateway) handles traffic routing. Before forwarding a request to an agent, it calls AgentShield's policy check endpoint to get an allow/deny decision.

URL Pattern

POST https://agentshield-api-622662891364.us-central1.run.app/api/v1/gateway/policy/check

Request Format

{
  "agentSlug": "finance-agent",        ← Target agent (or use workflowSlug)
  "user": {                             ← Caller identity (optional — falls back to JWT user)
    "id": "user-123",
    "role": "analyst",
    "email": "analyst@company.com",
    "department": "engineering"
  },
  "action": "invoke",                   ← Action being performed
  "customContext": {                    ← Additional context for policy conditions
    "source_ip": "10.0.0.50",
    "client_app": "trading-dashboard"
  }
}

Response Format

✅ Allowed
{
  "success": true,
  "data": {
    "allowed": true,
    "reason": "Policy \"Allow All Agents\" (allow)",
    "matchedPolicy": "Allow All Agents",
    "checkedAt": "2026-06-29T01:29:33.454Z",
    "target": {
      "type": "agent",
      "slug": "finance-agent",
      "name": "Finance Agent"
    }
  }
}
❌ Denied
{
  "success": true,
  "data": {
    "allowed": false,
    "reason": "Policy \"Block Finance Agent\" (deny)",
    "matchedPolicy": "Block Finance Agent",
    "checkedAt": "2026-06-29T01:29:33.454Z",
    "target": {
      "type": "agent",
      "slug": "finance-agent",
      "name": "Finance Agent"
    }
  }
}

Complete Example

# Check if a "viewer" role user can access finance-agent
curl -s -X POST "$API/gateway/policy/check" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "agentSlug": "finance-agent",
    "user": { "role": "viewer", "department": "marketing" },
    "action": "invoke"
  }' | python3 -m json.tool
⚠️ Important: Default Policy Behavior If no policies exist at all, the gateway allows all requests (default-allow when zero policies).
If policies exist but none match the request, it defaults to deny. This means you typically need a low-priority "Allow All" catch-all policy as a baseline.

04 Governance Pipeline Deep-Dive

Every request through the Full Gateway passes through 8 governance layers in sequence. If any layer rejects, the request stops immediately.

1

Authentication

Validates JWT token or API key. Extracts user identity (id, role, email, department).

❌ Fails → HTTP 401 Unauthorized
2

Agent Lookup

Resolves the agent-slug from the registry. Checks if agent is active and healthy.

❌ Fails → HTTP 503 "Agent inactive/unhealthy"
3

Policy Evaluation

Runs all active access_control policies in priority order. First match wins. Evaluates subjects (user role/email), resources (agent slug), and conditions (time, MFA).

❌ Fails → HTTP 403 "Access denied by policy"
4

Budget Check

Verifies spending limits for the user/team/agent. Checks both hard limits (block) and soft limits (warn).

❌ Fails → HTTP 402 "Budget exceeded"
5

Input Guardrails

Scans the request payload for prompt injection, jailbreak attempts, and other malicious input patterns.

❌ Fails → HTTP 400 "Input blocked by guardrails"
6

Agent Proxy

Forwards the request to the actual agent endpoint. Supports REST (POST) and MCP (SSE + JSON-RPC) protocols.

❌ Fails → HTTP 502 "Agent unavailable"
7

Output Guardrails

Scans the agent's response for PII (SSN, email, phone, credit card), profanity, PHI (HIPAA), and other sensitive data.

❌ Fails → Response sanitized or blocked
8

Cost Tracking & Audit

Records token usage, calculates cost based on model pricing, logs the full invocation to the immutable audit trail.

✅ Always runs (non-blocking)

06 Policy Configuration & Testing

Policy Evaluation Rules

  • Policies are evaluated in priority order (lower number = higher priority)
  • First matching policy wins (stop evaluation)
  • If no policy matches → default deny
  • If no policies exist at all → default allow

Policy JSON Schema

{
  "name": "Policy Name",
  "description": "Human-readable description",
  "policyType": "access_control",
  "priority": 10,                    ← Lower = higher priority
  "rulesJson": {
    "effect": "deny",                ← "allow" or "deny"
    "subjects": [                    ← WHO (user conditions)
      { "field": "role", "operator": "equals", "value": "viewer" }
    ],
    "resources": [                   ← WHAT (agent conditions)
      { "field": "slug", "operator": "equals", "value": "finance-agent" }
    ],
    "conditions": [                  ← WHEN (additional conditions)
      { "field": "time.hour", "operator": "greaterThan", "value": "18" }
    ]
  }
}

12 External Gateway Integration

When using AgentShield as a PDP alongside existing API gateways, integrate using these patterns:

Kong (Lua Plugin)

-- kong plugin: agentshield-policy-check
local http = require("resty.http")
local cjson = require("cjson")

function AgentShieldHandler:access(conf)
    local httpc = http.new()
    local res, err = httpc:request_uri(conf.agentshield_url .. "/api/v1/gateway/policy/check", {
        method = "POST",
        body = cjson.encode({
            agentSlug = kong.request.get_header("X-Agent-Slug"),
            user = {
                id = kong.client.get_consumer().id,
                role = kong.client.get_consumer().custom_id,
            },
            action = "invoke"
        }),
        headers = {
            ["Content-Type"] = "application/json",
            ["Authorization"] = "Bearer " .. conf.service_token,
        },
    })

    local decision = cjson.decode(res.body)
    if not decision.data.allowed then
        return kong.response.exit(403, {
            message = "Blocked by AgentShield: " .. decision.data.reason
        })
    end
end

AWS API Gateway (Lambda Authorizer)

// Lambda authorizer — calls AgentShield policy/check
exports.handler = async (event) => {
    const agentSlug = event.pathParameters?.agentSlug || 'unknown';
    const userRole = event.requestContext?.authorizer?.claims?.role || 'viewer';

    const response = await fetch(
        process.env.AGENTSHIELD_URL + '/api/v1/gateway/policy/check',
        {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'Authorization': `Bearer ${process.env.AGENTSHIELD_TOKEN}`,
            },
            body: JSON.stringify({
                agentSlug,
                user: { role: userRole },
                action: 'invoke',
            }),
        }
    );

    const decision = await response.json();

    return {
        isAuthorized: decision.data?.allowed === true,
        context: {
            policyDecision: decision.data?.reason,
            matchedPolicy: decision.data?.matchedPolicy,
        },
    };
};

13 SDK & Client Library Examples

Python

import requests

class AgentShieldClient:
    def __init__(self, base_url, email, password):
        self.base_url = base_url.rstrip('/')
        self.session = requests.Session()
        # Login
        resp = self.session.post(f"{self.base_url}/auth/login",
            json={"email": email, "password": password})
        self.token = resp.json()["data"]["token"]
        self.session.headers["Authorization"] = f"Bearer {self.token}"

    def invoke(self, agent_slug, prompt, **kwargs):
        """Invoke an agent through the full governance gateway"""
        payload = {"prompt": prompt, **kwargs}
        resp = self.session.post(
            f"{self.base_url}/gateway/agents/{agent_slug}/invoke",
            json=payload)
        return resp.json()

    def check_policy(self, agent_slug, user_role="viewer"):
        """Check if an action would be allowed (sidecar mode)"""
        resp = self.session.post(
            f"{self.base_url}/gateway/policy/check",
            json={"agentSlug": agent_slug, "user": {"role": user_role}})
        return resp.json()["data"]

# Usage
client = AgentShieldClient(
    "https://agentshield-api-622662891364.us-central1.run.app/api/v1",
    "admin@agentshield.local", "admin123")

# Full gateway mode
result = client.invoke("finance-agent", "Analyze Q3 revenue")
print(result["data"]["response"])

# Sidecar mode
decision = client.check_policy("finance-agent", user_role="viewer")
print(f"Allowed: {decision['allowed']}, Reason: {decision['reason']}")

JavaScript / Node.js

class AgentShieldClient {
    constructor(baseUrl) {
        this.baseUrl = baseUrl.replace(/\/$/, '');
        this.token = null;
    }

    async login(email, password) {
        const resp = await fetch(`${this.baseUrl}/auth/login`, {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ email, password }),
        });
        const data = await resp.json();
        this.token = data.data.token;
    }

    async invoke(agentSlug, prompt, options = {}) {
        const resp = await fetch(
            `${this.baseUrl}/gateway/agents/${agentSlug}/invoke`,
            {
                method: 'POST',
                headers: {
                    'Content-Type': 'application/json',
                    'Authorization': `Bearer ${this.token}`,
                },
                body: JSON.stringify({ prompt, ...options }),
            }
        );
        return resp.json();
    }

    async checkPolicy(agentSlug, userRole = 'viewer') {
        const resp = await fetch(`${this.baseUrl}/gateway/policy/check`, {
            method: 'POST',
            headers: {
                'Content-Type': 'application/json',
                'Authorization': `Bearer ${this.token}`,
            },
            body: JSON.stringify({
                agentSlug,
                user: { role: userRole },
            }),
        });
        return resp.json();
    }
}

14 Troubleshooting

IssueHTTP CodeCauseFix
"No matching policy — default deny" 403 Policies exist but none match Create a baseline "Allow All" policy with priority 1000
"Unauthorized" on gateway call 401 Missing or expired JWT Re-login via POST /auth/login
"Agent is currently inactive" 503 Agent deactivated in registry Activate via PUT /agents/{slug} with is_active: true
"Agent is currently unhealthy" 503 Health check failing Check agent's /health endpoint directly. Possible cold start.
"Budget exceeded" 402 Spending limit reached Delete or increase budget via DELETE /budgets/{id}
Empty or error response from agent 200 Gemini model deprecated or key missing Update GEMINI_MODEL env var on Cloud Run agent
"Cannot POST /invoke" 404 Calling agent's wrong endpoint Our agents use POST / (root), not /invoke
